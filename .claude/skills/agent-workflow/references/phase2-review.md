# 阶段2：设计审查 - 详细指南

## 负责代理
- `unity-design-reviewer` - 参考 `.claude/agents/unity-design-reviewer.md`
- `unity-performance` - 参考 `.claude/agents/unity-performance.md`（性能审查）

## 目标
通过系统化的审查和质询，发现设计方案中的问题、风险和改进空间，**特别关注性能设计的合理性**。

## 执行步骤

### 1. 理解设计（5分钟）
- 仔细阅读设计文档
- 理解设计的背景和目标
- 识别关键设计决策
- 绘制自己的理解图（验证理解）

### 2. 系统化审查（15-20分钟）

#### 2.1 架构完整性审查
- [ ] 是否覆盖所有功能需求？
- [ ] 模块职责是否清晰？
- [ ] 依赖关系是否合理？
- [ ] 是否存在循环依赖？
- [ ] 是否有遗漏的模块？

#### 2.2 设计质量审查
- [ ] 设计模式应用是否合适？
- [ ] 是否存在过度设计？
- [ ] 接口定义是否清晰？
- [ ] 数据流是否合理？
- [ ] 是否符合SOLID原则？

#### 2.3 可实施性审查
- [ ] 技术选型是否可行？
- [ ] 实施复杂度是否可控？
- [ ] 团队能力是否匹配？
- [ ] 是否有明确实施路径？
- [ ] 迁移策略是否清晰？

#### 2.4 性能设计审查⚡（unity-performance主导）
**性能目标审查**：
- [ ] 是否设定了明确的性能目标？
- [ ] 性能目标是否合理可达？
- [ ] 是否考虑目标平台的性能限制？

**性能关键路径审查**：
- [ ] 是否识别了所有性能关键路径？
- [ ] 性能敏感操作是否标记清楚？
- [ ] 是否评估了潜在瓶颈？

**性能优化策略审查**：
- [ ] 对象池设计是否合理？池大小是否合适？
- [ ] 缓存策略是否有效？是否会过度缓存？
- [ ] 异步设计是否避免了主线程阻塞？
- [ ] 内存管理策略是否完善？

**性能风险评估**：
- [ ] 是否识别了性能风险？
- [ ] 风险等级是否合理？
- [ ] 应对方案是否可行？

**常见性能问题检查**：
- [ ] 是否避免在Update中进行昂贵操作？
- [ ] 是否有计划缓存组件引用？
- [ ] 是否考虑了GC分配问题？
- [ ] 是否有Draw Call优化策略？（如适用）
- [ ] 是否考虑了内存泄漏预防？

#### 2.5 扩展性审查
- [ ] 是否支持功能扩展？
- [ ] 扩展点设计是否合理？
- [ ] 是否便于添加新功能？

#### 2.6 风险审查
- [ ] 是否引入新技术债务？
- [ ] 是否考虑边界情况？
- [ ] Unity特定问题是否处理？
- [ ] 是否有应对方案？

### 3. 提出质询（10-15分钟）

#### 质询方法

**5W1H分析**：
- What: 这个设计具体要解决什么问题？
- Why: 为什么选择这种方案？
- Who: 谁来实施和维护？
- When: 什么时候实施？分几个阶段？
- Where: 在哪些模块应用？
- How: 具体如何实施？

**反向思维**：
- 如果失败，最可能的原因是什么？
- 哪些假设不成立会导致失效？
- 最坏情况是什么？如何应对？

**边界探索**：
- 性能极限在哪里？
- 扩展性边界在哪里？
- 什么情况下会失效？

### 4. 质询问题示例

#### 架构层面
1. 为什么选择这种架构模式？考虑过其他方案吗？
2. 规模扩大10倍时还能工作吗？
3. 需求重大变化时，适应性如何？
4. 模块间通信是否高效？

#### 设计模式层面
1. 使用这个模式的真正原因是什么？
2. Unity环境下有特殊问题吗？
3. 是否增加了不必要的复杂度？
4. 有更简单的方案吗？

#### 性能层面⚡（unity-performance质询）
**性能目标**：
1. 为什么选择这个性能目标？是否合理？
2. 如何验证性能目标？有测试计划吗？
3. 如果性能不达标，备用方案是什么？

**性能设计**：
1. 性能关键路径是如何识别的？有遗漏吗？
2. 对性能有什么影响？有预估数据吗？
3. 是否存在频繁的内存分配？如何减少？
4. 对象池大小是如何确定的？依据是什么？
5. 为什么这里使用缓存？缓存失效如何处理？
6. 异步操作会不会影响帧率？

**性能风险**：
1. 最大的性能风险是什么？
2. 在低端设备上表现如何？有测试计划吗？
3. 如果同时有1000个对象，性能如何？
4. 内存占用会增加多少？是否可接受？

**Unity特定性能**：
1. 是否考虑了GC分配问题？如何避免？
2. Draw Call预计会增加多少？（如适用）
3. 是否需要LOD或遮挡剔除？（如适用）
4. 物理计算对性能的影响？（如适用）
3. 低端设备上表现如何？
4. 是否考虑对象池、缓存等优化？

#### 实施层面
1. 需要多少工作量？有评估吗？
2. 是否需要大规模重构？
3. 团队是否都理解这个设计？
4. 如何保证实施质量？

### 6. 与架构师讨论（5-10分钟）
- 提出问题，听取解释
- 讨论改进方案
- 达成共识或标记分歧
- 决定是否需要调整设计

## 审查态度原则

### 保持专业
- 客观公正，基于事实
- 尊重设计者的工作
- 目标是改进，非批评

### 追求完善
- 不放过可疑之处
- 深入挖掘潜在问题
- 考虑极端情况

### 建设性沟通
- 提问题，同时给建议
- 承认设计的优点
- 愿意讨论和妥协

## 审查结论判断

### ✅ 通过
- 没有高优先级问题
- 中低优先级问题可接受
- 设计合理可行

### ⚠️ 有条件通过
- 有中优先级问题需解决
- 需要小幅调整
- 总体方向正确

### ❌ 不通过
- 有高优先级问题
- 架构方向有误
- 需要重新设计
